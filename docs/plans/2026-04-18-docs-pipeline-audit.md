# Docs Pipeline Audit — Phase 1

*Date: 2026-04-18*
*Scope: shared/scripts/\*generator.py + docs/ rendering surface*
*Status: audit only; no code changes in this memo*

## 1. Purpose

The adk-fluent documentation pipeline has five generators
(`doc_generator.py`, `llms_generator.py`, `skill_generator.py`,
`concepts_generator.py`, `cookbook_generator.py`, `readme_generator.py`)
that together produce ~60 generated markdown files, a `llms.txt`
surface, six editor-specific context files, and an injected
`architecture-and-concepts.md` stub. They all read from the same
upstream pair — `shared/manifest.json` (scanned ADK surface) and
`shared/seeds/seed.toml` (hand-curated overlay) — but each generator
re-implements its own view of "builder inventory", "namespace methods",
and "pattern catalogue". Drift is inevitable and already observable.

This memo is the Phase 1 output of the Pipeline Overhaul brief: a file
ownership inventory, a dependency graph, a drift catalogue, and a
`DocIR` Pydantic schema proposal. No generator behaviour changes.

## 2. File ownership inventory

The `docs/` tree contains 259 markdown/rst files as of this audit.
Ownership breaks down into five classes:

### 2.1 Generated by `doc_generator.py` (~60 files)

- `docs/generated/builders/*.md` — one per builder × module
  (e.g. `agent/Agent.md`, `workflow/Pipeline.md`).
- `docs/generated/namespaces/{p,c,s,a,m,t,g,ui}.md` — one per namespace,
  produced from the hand-declared `NAMESPACE_MODULES` list at
  `shared/scripts/doc_generator.py:67-135`.
- `docs/generated/cookbook/index.md` — categorised index produced by
  `_categorize_cookbook` at `shared/scripts/doc_generator.py:1317-1336`.
- `docs/generated/cookbook/*.md` — one per `examples/cookbook/*.py`,
  *unless* the target file matches the "preserve if hand-crafted"
  heuristic at `shared/scripts/doc_generator.py:1735-1739`
  (`"## Layer "`, `"## Pattern"`, or `"Pathway:"` anywhere in the
  existing file).

### 2.2 Generated by `llms_generator.py` (8 files)

- `docs/llms.txt` — the canonical agent-readable surface.
- `.cursorrules`, `.github/copilot-instructions.md`, `.zed/.rules`,
  `CLAUDE.md` — editor-specific wrappers produced by
  `_wrap_for_claude_md`, `_wrap_for_cursor`, `_wrap_for_copilot`,
  `_wrap_for_zed` at `shared/scripts/llms_generator.py:1131-1164`.
- Three more auxiliary context files emitted from the same run.

### 2.3 Generated by `concepts_generator.py` (1 file + 1 injection)

- `docs/user-guide/architecture-and-concepts.md` — extracted by string
  matching from `docs/other_specs/adk_fluent_v51_state_thesis.md`
  and `adk_fluent_v51_context.md`.
- `docs/user-guide/index.md` — mutated in-place: line 77 does
  `index_content.replace(":maxdepth: 2\n", ":maxdepth: 2\n\narchitecture-and-concepts")`
  guarded only by a string-contains check, which is not robust against
  edits to the toctree header format.

### 2.4 Copied via `html_extra_path` (7 files)

Listed in `docs/conf.py:211-220`:

- `pcs-visual-reference.html`, `operator-algebra-reference.html`,
  `module-lifecycle-reference.html`, `data-flow-reference.html`,
  `delegation-reference.html`, `execution-modes-reference.html`,
  `a2a-topology-reference.html`, plus `llms.txt`.
- These bypass Sphinx entirely; no MyST parsing, no cross-reference
  validation, no link checking.

### 2.5 Hand-written (~191 files)

Everything else: the landing page (`docs/index.md`), getting-started,
user-guide chapters (other than the injected one), design notes under
`docs/plans/`, and concept specs under `docs/other_specs/` (the latter
excluded from the Sphinx build via `conf.py:203`).

Two additional, disconnected subgraphs:

- **`concepts_generator.py`** reads only
  `docs/other_specs/adk_fluent_v51_state_thesis.md` and
  `adk_fluent_v51_context.md`. It does *not* read manifest.json or
  seed.toml. Fragile because its inputs are hand-authored long-form
  specs.
- **`readme_generator.py`** reads `README.template.md` +
  `CHANGELOG.md`, imports the live `adk_fluent` package to render a
  pipeline mermaid diagram (`shared/scripts/readme_generator.py:26-62`),
  and writes both `README.md` and `python/README.md`.

`cookbook_generator.py` is a fifth consumer that shares the
`generator/` subpackage (L36) but writes Python files, not markdown.

## 4. Drift catalogue

Problems observed in the pipeline, each with a file:line citation. Items
are independent — fixing one does not force fixing another.

### 4.1 Triple-rendering of the builder inventory

The same conceptual "builder table" is rendered by three generators
with three independent code paths:

- `doc_generator.py` — table emission embedded in per-namespace
  template code.
- `llms_generator.py:51-60` — `_format_builder_table` produces a
  Markdown table of builder name + one-liner.
- `skill_generator.py:301` — `generate_builder_inventory` produces a
  third variant for the skills package.

Any shape change to the underlying builder record (e.g. new
`deprecated` flag, renamed `source_class_short`) requires a coordinated
three-file patch or the outputs disagree.

### 4.2 Lazy cross-generator imports

`skill_generator.py` imports helpers from `llms_generator` inside
functions at lines 880 and 1012 rather than at module level. This
hides the coupling from any static analyser and means a rename in
`llms_generator.py` fails only at runtime of a specific `just skills`
invocation.

### 4.3 Hand-declared namespace list

`shared/scripts/doc_generator.py:67-135` hard-codes six
`NamespaceSpec` entries (P, C, S, A, M, T). The actual runtime now has
ten namespaces when you count G, H, E, UI. Adding a namespace requires
editing this constant, the corresponding section of
`llms_generator.py`, and `skill_generator.py`'s own `generate_namespace_methods`
at line 393. Drift already visible: the generated namespace docs page
does not list UI even though `adk_fluent.UI` is part of the public
surface.

### 4.4 Fragile heading extraction in concepts_generator

`shared/scripts/concepts_generator.py` extracts content by exact
string matching on H2/H1 boundaries:

- L31 `if line.startswith("## The Three Channels"):` — if the spec
  author renames the heading, the file silently becomes a 4-line stub.
- L57 `if "| Operation |" in line:` — table header string match; a
  reordered column produces silent empty output.
- No validation checks that *any* content was extracted.

### 4.5 Unguarded TOC injection

`concepts_generator.py:77` does:

```python
index_content.replace(":maxdepth: 2\n", ":maxdepth: 2\n\narchitecture-and-concepts")
```

This is guarded only by `if "architecture-and-concepts" not in
index_content:` (line 75). If the `user-guide/index.md` toctree header
is rewritten to `:maxdepth: 3`, to omit the blank line, or to use
different whitespace, the replacement becomes a no-op but the
idempotency check still passes on subsequent runs — the file is never
updated again and the link is permanently missing.

### 4.6 Orphaned spec files

`docs/other_specs/` contains 9 historical specs
(`adk_fluent_v3_spec.docx`, `adk_fluent_v4_spec.md`,
`adk_fluent_v5_spec.md`, `adk_fluent_v51_spec.md`, plus appendices).
Only two are read by any generator (`concepts_generator.py`). The rest
are excluded from the Sphinx build (`conf.py:203`) and unreachable from
the docs tree. They are neither source-of-truth nor rendered output —
they drift freely.

### 4.7 Mixed generator/injector responsibility

`concepts_generator.py` both *writes a file* (L69) and *mutates an
unrelated hand-edited file* (L78). Two distinct responsibilities in
one script makes it impossible to run one without the other or to
dry-run either safely.

### 4.8 `html_extra_path` bypasses validation

Seven HTML reference pages copy straight into the build output via
`conf.py:211-220`. They are not linted, not link-checked, and not
cross-referenced by Sphinx. Anchors within them are invisible to
`:ref:` and `hoverxref`.

### 4.9 Preserve-if-hand-crafted heuristic

`doc_generator.py:1735-1739` decides to skip regeneration of a
cookbook file if any of three substrings appear in the existing file.
This is unprincipled:

- A hand-written file that happens not to contain any of those
  three markers is silently clobbered.
- A generated file that a reviewer lightly edited to add "## Pattern"
  as a heading becomes locked against future regeneration without
  warning.

The heuristic has no marker comment distinguishing "this file is
hand-owned" from "this file is generated".

### 4.10 Duplicate title-case normalisation

`_anchor_id` at `doc_generator.py:143-145` normalises names for
Markdown anchors. `skill_generator.py` re-implements a near-identical
normalisation. `llms_generator.py` uses a third variant for
`llms.txt` section anchors. A builder named `MCPTool` produces three
anchor casings.

### 4.11 Hard-coded cookbook category ranges

`doc_generator.py:1317-1336` maps numeric file prefixes to category
names (`1-7 Basics`, `8-13 Execution`, ...). Adding a cookbook at
index `75+` silently falls into "Skills & Harness" even if topically
unrelated. The category boundaries are not declared in seed.toml.

### 4.12 `index.md` mutation without backup

`concepts_generator.py:78` overwrites `user-guide/index.md` in place
with no `.bak` or git-diff surface prior to write. A malformed
injection has no rollback path short of `git checkout`.

### 4.13 No validation layer

Nothing in the pipeline validates:

- internal links (`[text](path.md#anchor)`) resolve,
- Sphinx `:ref:` targets exist,
- every public builder has a docs page,
- every cookbook file has a matching generated markdown entry,
- the namespace method list matches the runtime-exported attributes.

The check that exists (`just check-gen`) only asserts byte-equality
between tracked outputs and a re-run — it catches staleness but not
correctness.

### 4.14 Additional finding — no DocIR

All five generators walk the same `BuilderSpec`/`ManifestClass`
Python structures independently. There is no *intermediate* document
representation that could be emitted, diffed, cached, or consumed by
external tooling. A new generator (e.g. for an alternate theme, PDF,
or API explorer) must re-implement the traversal from scratch.

## 5. DocIR proposal

A typed intermediate representation that all generators consume.
Proposed shape (Pydantic v2, sketch only):

```python
class BuilderRef(BaseModel):
    name: str                         # e.g. "Agent"
    module: str                       # e.g. "agent"
    source_class: str | None          # e.g. "google.adk.agents.LlmAgent"
    is_composite: bool = False
    is_standalone: bool = False
    aliases: dict[str, str]           # fluent_name -> field_name
    constructor_args: list[str]
    fields: list[FieldRef]
    description: str | None
    anchor_id: str                    # canonical, computed once

class NamespaceDoc(BaseModel):
    symbol: str                       # "P", "C", "S", ...
    title: str                        # "Prompt composition"
    module: str                       # "adk_fluent._prompt"
    methods: list[NamespaceMethod]
    composition_operators: list[str]  # ["+", "|"]

class NamespaceMethod(BaseModel):
    name: str                         # "role"
    signature: str                    # "P.role(text: str) -> P"
    summary: str                      # one line
    description: str | None           # full body
    examples: list[CodeBlock]

class CookbookEntry(BaseModel):
    index: int                        # numeric prefix
    slug: str                         # filename without ".py"
    title: str
    category: str                     # pulled from seed, not hard-coded
    source_path: Path
    render_path: Path
    preserve: bool = False            # explicit marker, not heuristic

class ConceptSection(BaseModel):
    slug: str                         # "three-channels"
    title: str
    body: str                         # Markdown source
    provenance: ProvenanceRef         # source file + heading

class DocIR(BaseModel):
    adk_version: str
    scan_timestamp: str
    builders: list[BuilderRef]
    namespaces: list[NamespaceDoc]
    cookbooks: list[CookbookEntry]
    concepts: list[ConceptSection]
    operators: list[OperatorRef]
    patterns: list[PatternRef]
```

With DocIR in place:

- `doc_generator`, `llms_generator`, `skill_generator` all become
  pure renderers over a typed input — they stop re-reading
  `manifest.json` and `seed.toml` independently.
- `cookbook_generator` writes CookbookEntry records, not raw Python.
- `concepts_generator` emits ConceptSection, not a pre-formatted
  Markdown file; TOC injection becomes a DocIR consumer concern.
- A new `docs_validator.py` can walk `DocIR` to check that every
  builder has a page, every method is documented, every cookbook
  referenced, every anchor unique.

The one-time cost is a `build_doc_ir.py` that replaces the four
scattered `parse_seed/parse_manifest/resolve_builder_specs` call
sites. The recurring benefit is that every rendering divergence
between generators becomes a schema-level observation rather than a
grep-across-five-files investigation.

## 6. Recommendation

Stop after this memo. The safe changes queued for Phase 2 without
touching byte-output of any generator are:

1. Harden `concepts_generator.py` — raise explicit errors when a
   heading or table is not found; make the TOC injection idempotent
   on file shape, not just on substring presence.
2. Add `docs_validator.py` — a standalone linter over the rendered
   `docs/generated/` tree checking for unresolved links and dangling
   anchors. Does not modify any generator output.
3. Add `just validate-docs` — wires the validator into the existing
   justfile, behind its own target so it cannot break `just docs`.

Phase 3 (DocIR introduction) is intentionally deferred until these
validation rails exist — otherwise the schema change has no
regression safety net.
